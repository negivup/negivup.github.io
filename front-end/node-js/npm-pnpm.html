pnpm 简介 #

简介 #

pnpm 是 Node.js 的一款包管理工具，可以认为是 Performant npm（高性能的 npm）的缩写。官网的描述 :

Fast, disk space efficient package manager

快速且省磁盘空间的包管理工具

快速

pnpm 比其他包管理工具（如 npm 和 yarn）快 2 倍。

高效

node_modules 中的文件复制或链接自特定的内容寻址存储库。

支持 monorepos

pnpm 内置支持单个仓库管理多个不同的包（monorepos）。

严格

pnpm 默认创建了一个非平铺的 node_modules，代码无法访问非项目 package.json 管理的包，也叫做幽灵依赖。

当使用 npm 或 yarn 时，如果有 100 个项目安装了同一个依赖（dependency），在磁盘上就会安装 100 个副本（一个依赖（dependency）占用 1M 的内存空间，100 个副本就是 100M）。

如果使用 pnpm，依赖会被存放在统一位置 :

  • 对同一个依赖使用相同的版本，在磁盘上只会存储一份文件
  • 对同一个依赖使用不同的版本，在磁盘上只会存储版本之间有差异的文件
  • 所有文件都会磁盘上同一个位置进行存储，项目中的文件会硬链接到实际文件所在的位置，不会额外占用磁盘空间

当使用 npm 或 yarn 时，所有依赖（dependency）都会被提升到 node_modules 的根目录下，因此代码可以访问不属于当前项目所管理的依赖（dependency）。

pnpm 默认使用软链接的方式将项目的直接依赖添加到 node_modules 中，其余依赖不放在 node_modules 根目录下。

pnpm 中所有文件都会磁盘上同一个位置进行存储，项目中的依赖会硬链接到实际文件所在的位置，幽灵依赖会以软链接的方式指向文件所在的实际位置。

pnpm 安装的依赖有如下的对应关系 :

  1. 在项目 package.json 中声明的依赖 : node_modules 中会管理依赖的软链接，在 node_modules/.pnpm 中找到依赖的硬链接
  2. 幽灵依赖 : 在 node_modules/.pnpm 中找到依赖的硬链接
  3. 幽灵依赖的依赖 : 会建立到缓存目录依赖的软链接

总结 : node_modules 中的文件夹是以软链接的方式存在，node_modules 中的文件是以硬链接的方式存在

硬链接和软链接 #

文件拷贝

文件的拷贝，会在磁盘中复制一份新的文件备份，两个文件相互独立 :

  • 修改 : 不会影响其他文件
    • 修改 foo.js，foo_copy.js 的内容不会变化
    • 修改 foo_copy.js，foo.js 的内容不会变化
  • 删除 : 不会影响其他文件
    • 删除 foo.js，foo_copy.js 仍会存在
    • 删除 foo_copy.js，foo.js 仍会存在
bash
# Windows
$ copy foo.js foo_copy.js

# 非 Windows
$ cp foo.js foo_copy.js

文件的硬链接

文件的硬链接，会在磁盘中生成一个文件备份，两个文件存在关联 :

  • 修改 : 会影响其他文件
    • 修改 foo.js，foo_hard.js 的内容会跟着变化
    • 修改 foo_hard.js，foo.js 的内容会跟着变化
  • 删除 : 不会影响其他文件
    • 删除 foo.js，foo_hard.js 仍会存在
    • 删除 foo_hard.js，foo.js 仍会存在
bash
# Windows
$ mklink /H foo_hard.js foo.js

# 非 Windows
$ ln foo.js foo_hard.js

文件的软链接

文件的软链接也叫符号链接，会在磁盘中创建一个文件的快捷方式 :

  • 修改 : 会影响其他文件
    • foo_soft.js 不可修改，修改之后快捷方式会失效
    • foo.js 可修改，修改之后不会影响软链接的指向
  • 删除 : 不会影响其他文件
    • 删除 foo.js，foo_soft.js 仍会存在，但软链接会找不到其对应的文件
    • 删除 foo_soft.js，foo.js 仍会存在，且文件内容不会变
bash
# Windows
$ mklink foo_soft.js foo.js

# 非 Windows
$ ln -s foo.js foo_soft.js
