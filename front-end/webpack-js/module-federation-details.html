插件详解 #

在 Webpack 中每个构建都是一个容器，模块联邦（Module Federation）中一个被引入的容器被称为 remote（对外暴露模块），引用者被称为 host（使用其他容器对外暴露的模块）。

从上图中可以看出，每个可以独立构建的程序都被称为一个容器。容器 1 引入容器 2 中的暴露的模块，模块 1 被称为 host，模块 2 被称为 remote。同时模块 2 引入模块 3 中暴露的模块，模块 2 被称为 host，模块 3 被称为 remote。

实现模块联邦（Module Federation）用到的插件主要有以下几个：

  • require('webpack').container.ContainerPlugin：该插件为指定的导出模块创建一个额外的容器入口
  • require('webpack').container.ContainerReferencePlugin：该插件将特定的容器添加为外部引用，并允许从这些容器中导入远程模块
  • require('webpack').sharing.SharePlugin：用于定义共享模块

一般情况下，配置模块联邦（Module Federation）时很少一次性使用上面提到的几个插件，而是使用一个组合插件：

  • require('webpack').container.ModuleFederationPlugin：包含 ContainerPlugin、ContainerReferencePlugin 和 SharePlugin

ContainerPlugin #

ContainerPlugin 插件的作用是为导出模块创建一个额外的容器入口，供 host 容器来调用。

插件参数
ts
declare interface ContainerPluginOptions {
  /**
   * Modules that should be exposed by this container. When provided, property name is used as public name, otherwise public name is automatically inferred from request.
   */
  exposes: Exposes;

  /**
   * The filename for this container relative path inside the `output.path` directory.
   */
  filename?: string;

  /**
   * Options for library.
   */
  library?: LibraryOptions;

  /**
   * The name for this container.
   */
  name: string;

  /**
   * The name of the runtime chunk. If set a runtime chunk with this name is created or an existing entrypoint is used as runtime.
   */
  runtime?: string | false;

  /**
   * The name of the share scope which is shared with the host (defaults to 'default').
   */
  shareScope?: string;
}

安装依赖：

bash
$ pnpm add --save-dev --save-exact webpack webpack-cli

devDependencies:
+ webpack 5.73.0
+ webpack-cli 4.9.2

创建打包的入口文件 src/index.js：

js
console.log('remote entry file ...');

1.name 和 exposes：

name 和 exposes 是必传参数，name 定义当前导出模块的全局变量名称，exposes 指定需要导出的模块，它是一个对象，key 一般使用相对路径的方式。

创建 Webpack 的配置文件 webpack.config.js：

js
const { ContainerPlugin } = require('webpack').container;

/** @type {import('webpack').Configuration} */
const webpackConfig = {
  entry: './src/index.js',
  output: { clean: true },
  plugins: [
    new ContainerPlugin({
      name: 'remoteApp',
      exposes: {
        './entry': './src/index.js',
      },
    }),
  ],
};

module.exports = webpackConfig;

为了能够正常查看打包后的代码，使用 pnpm run webpack --mode=none 打包。打包后的目录结构：

dist
|-- 2.js
|-- main.js
|-- remoteApp.js

其中 main.js 是当前程序打包后的文件，remoteApp.js 是当前容器导出的入口文件（供 host 容器使用的入口），那 2.js 是怎么来的呢？下面我们来看一下 remoteApp.js 中的代码，研究一下加载逻辑。

首先，name 指定的并不是打包后的文件名称，而是导出变量的名称，而没有指定 filename，会把 name 的值作为 filename 来使用，因此容器的入口文件名称是 remoteApp.js。remoteApp.js 的第一行就展示来 name 值就是最终代码中会挂载到 window 上的变量名称。

js
var remoteApp;

所有导出的模块会放在一个 moduleMap 对象中：

js
var moduleMap = {
  './entry': () => {
    return __webpack_require__.e(2).then(() => () => __webpack_require__(1));
  },
};

__webpack_require__.e 会被调用，这个函数的定义如下：

js
__webpack_require__.e = (chunkId) => {
  return Promise.all(
    Object.keys(__webpack_require__.f).reduce((promises, key) => {
      __webpack_require__.f[key](chunkId, promises);
      return promises;
    }, []),
  );
};

这个方法调用了 __webpack_require__.f 中的属性对应的方法，而 __webpack_require__.f.j 有一句核心代码 var url = __webpack_require__.p + __webpack_require__.u(chunkId)，最终把 chunkId 传递给了 __webpack_require__.u，查看这个方法的实现：

js
__webpack_require__.u = (chunkId) => {
  return '' + chunkId + '.js';
};

发现它的返回值就是 .js 文件的地址，而 chunkId 在上面传递的是 2，因此这里找到的就是 2.js。这就是打包结果中会出现一个 2.js 的原因。

2.filename

filename 的作用就是指定 remote 容器入口的文件名称，如果不指定，默认以 name 的值作为文件名。修改 webpack.config.js 增加 filename: 'remoteEntry.js'：

js
const { ContainerPlugin } = require('webpack').container;

/** @type {import('webpack').Configuration} */
const webpackConfig = {
  entry: './src/index.js',
  output: { clean: true },
  plugins: [
    new ContainerPlugin({
      name: 'remoteApp',
      filename: 'remoteEntry.js',
      exposes: {
        './entry': './src/index.js',
      },
    }),
  ],
};

module.exports = webpackConfig;

使用 pnpm run webpack --mode=none 打包。打包后的目录结构：

dist
|-- 2.js
|-- main.js
|-- remoteEntry.js

3.runtime

设置了 runtime 后，就会把运行时需要的代码单独打包成一个 chunk，随后当前容器导出的入口文件（供 host 容器使用的入口）就需要先使用这个文件，在加载入口文件才能正常工作。

提示

runtime 这个属性在模块联邦中使用的频率较小。

在配置中增加 runtime: 'remote-entry-runtime.js'：

js
const { ContainerPlugin } = require('webpack').container;

/** @type {import('webpack').Configuration} */
const webpackConfig = {
  entry: './src/index.js',
  output: { clean: true },
  plugins: [
    new ContainerPlugin({
      name: 'remoteApp',
      filename: 'remoteEntry.js',
      runtime: 'remote-entry-runtime.js',
      exposes: {
        './entry': './src/index.js',
      },
    }),
  ],
};

module.exports = webpackConfig;

使用 pnpm run webpack --mode=none 打包。打包后的目录结构：

dist
|-- 3.js
|-- main.js
|-- remote-entry-runtime.js
|-- remoteEntry.js

4.shareScope

shareScope 用于设置共享作用域的变量名称，默认是 default，可以在打包后的代码 remoteEntry.js 中找到：

js
var init = (shareScope, initScope) => {
  if (!__webpack_require__.S) return;
  // 在这里可以看到默认的名称
  var name = 'default';
  var oldScope = __webpack_require__.S[name];
  if (oldScope && oldScope !== shareScope)
    throw new Error(
      'Container initialization failed as it has already been initialized with a different share scope',
    );
  __webpack_require__.S[name] = shareScope;
  return __webpack_require__.I(name, initScope);
};

如果在配置文件 webpack.config.js 中指定了名称：

js
const { ContainerPlugin } = require('webpack').container;

/** @type {import('webpack').Configuration} */
const webpackConfig = {
  entry: './src/index.js',
  output: { clean: true },
  plugins: [
    new ContainerPlugin({
      name: 'remoteApp',
      filename: 'remoteEntry.js',
      runtime: 'remote-entry-runtime.js',
      shareScope: 'negivup',
      exposes: {
        './entry': './src/index.js',
      },
    }),
  ],
};

module.exports = webpackConfig;

使用 pnpm run webpack --mode=none 打包，查看打包后的 remoteEntry.js，发现名称变成了指定的值。

js
var init = (shareScope, initScope) => {
  if (!__webpack_require__.S) return;
  var name = 'negivup';
  var oldScope = __webpack_require__.S[name];
  if (oldScope && oldScope !== shareScope)
    throw new Error(
      'Container initialization failed as it has already been initialized with a different share scope',
    );
  __webpack_require__.S[name] = shareScope;
  return __webpack_require__.I(name, initScope);
};

5.library

library 用于指定打包后文件的模块组织方式（默认值是 var），可以通过下面的方式打包代码，在最终目录中查看不同类型代码的区别：

需要说明的是，library.type 是必须要指定的，library.name 在特定的情况下才需要。

js
const { ContainerPlugin } = require('webpack').container;

module.exports = [
  // 默认值
  'var',
  // 'module',
  'assign',
  'assign-properties',
  'this',
  'window',
  'self',
  'global',
  'commonjs',
  'commonjs2',
  // 'commonjs-module',
  'commonjs-static',
  // 'amd',
  // 'amd-require',
  'umd',
  'umd2',
  'jsonp',
  'system',
].map((type) => {
  /** @type {import('webpack').Configuration} */
  return {
    entry: './src/index.js',
    output: { clean: true },
    plugins: [
      new ContainerPlugin({
        name: 'remoteApp',
        filename: `${type}.js`,
        exposes: { './entry': './src/index.js' },
        library: { type, name: 'remoteAppLibrary' },
      }),
    ],
  };
});

提示

由于模块联邦的基础是多个应用独立部署，因此常常会出现跨域的问题。为了解决跨域问题，最好使用将 library.type 设置为 jsonp，不要忘了设置 library.type.name 否则无法正常打包。

ContainerReferencePlugin #

ContainerReferencePlugin 插件的作用是加载 remote 容器的入口文件。

插件参数
ts
declare interface ContainerReferencePluginOptions {
  /**
   * The external type of the remote containers.
   */
  remoteType: ExternalsType;

  /**
   * Container locations and request scopes from which modules should be resolved and loaded at runtime. When provided, property name is used as request scope, otherwise request scope is automatically inferred from container location.
   */
  remotes: Remotes;

  /**
   * The name of the share scope shared with all remotes (defaults to 'default').
   */
  shareScope?: string;
}

使用这个插件需要有 remote 容器存在，因此需要创建两个项目，一个是 host 容器，一个是 remote 容器。为了方便管理依赖，这里采用和简单使用中类似的方式管理（使用 pnpm 作为包管理工具，用 workspace 管理 Monorepo）。

直接下载基础项目代码，安装依赖，运行程序就可以看到效果：

bash
$ pnpm install
# 打开两个控制台，同时运行下面两个命令
$ pnpm run app1:dev
$ pnpm run app2:dev

依赖的版本如下：

bash
devDependencies:
